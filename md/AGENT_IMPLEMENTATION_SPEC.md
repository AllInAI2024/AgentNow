# AgentNow 企业智能体功能实现方案

> 版本：v1.1
> 日期：2026-04-29
> 状态：开发前定稿
> 适用范围：AgentNow 基于 Hermes 的企业端智能体能力建设

---

## 一、目标与约束

### 1.1 目标

在 AgentNow 现有“权限管理 + Hermes 系统概览 + 知识库管理”基础上，新增企业员工可直接使用的智能体能力，实现以下闭环：

1. 一个企业可以为每个员工分配一个 Hermes 智能体
2. 每个员工的工作记忆彼此隔离
3. 企业公共知识可被所有员工智能体共享
4. 平台管理员可统一配置岗位智能体模板和知识库
5. 先落地一个高价值案例：**帮助员工写 PPT 的智能体**

### 1.2 必须遵守的约束

结合 `md/SPEC.md` 与现有设计，智能体方案必须遵守下面几条：

1. **不改造 Hermes**，仅通过 Hermes 原生 CLI、Profile、API Server、MCP、记忆与 RAG 能力集成
2. **AgentNow 只做企业封装层**，负责账号、组织、权限、配置、审计、前端交互、路由编排
3. **记忆隔离依赖 Hermes Profile**，不是在 AgentNow 自己重复实现一套记忆系统
4. **知识共享通过共享知识库目录/MCP 能力实现**，不是把知识复制到每个员工 Profile
5. **首期优先保证“可用闭环”**，而不是一次把多岗位、多模态、多流程全部做完

### 1.3 文档分工

为了避免开发阶段“看错文档、按错版本”。

当前 Agent 相关文档建议按下面方式使用：

1. `AGENT_IMPLEMENTATION_SPEC.md`
   - 用来说明总体目标、边界、架构、模块、接口和实施路径
   - 它回答“整体要做什么、为什么这样做”

2. `AGENT_PPT_ASSISTANT_TEMPLATE_V1.md`
   - 用来定义 `ppt_assistant` 这个首个模板怎么配置、怎么交互、怎么输出
   - 它回答“PPT 助手这个模板具体怎么工作”

3. `AGENT_IMPLEMENTATION_TASKS.md`
   - 用来指导正式开发时一步一步推进
   - 它回答“按什么顺序开发、每一步怎么交代给大模型”

4. `AGENT_API_SPEC.md`
   - 用来定义员工侧接口、管理端接口、请求参数、返回结构、状态枚举和错误口径
   - 它回答“接口怎么设计、前后端怎么联调”

补充一个单一事实来源：

5. `backend/data/database.sql`
   - 作为当前数据库结构的最终落地版本
   - 如果文档中的字段示意和 SQL 不一致，以 SQL 为准，再回头修文档

补充一份术语对齐文档：

6. `AGENT_GLOSSARY.md`
   - 统一 Hermes / Hermes WebUI / AgentNow 的关键概念对照
   - 用于解决“Agent 到底是什么、模板/技能/工具怎么分工”的口径一致性问题

### 1.4 术语与概念对齐（非常重要）

在 Hermes 体系中，“Agent”更偏运行时系统本体，它通过 Profile、Memory、Skills、Tools、Session 等组合来体现能力差异。

而在 AgentNow 体系中，“智能体（Agent）”更偏企业产品对象，它必须同时满足：

1. 可配置（模板可运营）
2. 可发放（员工实例可治理）
3. 可追踪（会话阶段、确认点、结构化结果可见）
4. 可交付（文件产物可下载、可版本化）
5. 可审计（权限与操作日志可回溯）

因此第一版的核心设计不是“在平台里再造一个 Hermes Agent”，而是：

- 用 Hermes 承担推理、记忆、工具调用运行时
- 用 AgentNow 承担企业侧模板、权限、审计、业务闭环与交付

完整术语表见 `md/AGENT_GLOSSARY.md`。

## 二、现状评估

### 2.1 已有能力

当前系统已经具备以下基础：

1. **员工与权限体系**
   - 已有员工管理、部门管理、角色权限管理
   - `users` 表已存在 `hermes_profile`、`hermes_profile_config` 字段，可承载 Hermes 映射

2. **Hermes 管理能力**
   - 已有 Hermes 系统概览、配置、记忆、技能、MCP、知识库、审计等管理页面与接口基础
   - `backend/app/services/hermes_service.py` 已能读取 Hermes 安装、Profile 目录、记忆文件、配置等信息

3. **知识库基础设施**
   - 已落地知识库文档管理
   - 文档统一存储在共享目录
   - Hermes 已规划通过 MCP 访问共享知识库

### 2.2 当前缺口

真正“员工可用的智能体能力”还缺下面几层：

1. 员工侧智能体列表与进入入口
2. 员工首次使用时自动开通 Hermes Profile
3. Agent 模板与岗位 Prompt/SOUL 管理
4. 对话会话管理、流式响应、历史记录
5. 员工只能访问自己的 Profile，对其他员工 Profile 完全不可见
6. 公共知识库如何被所有员工智能体稳定调用
7. 首个业务案例的模板、知识准备、验收标准

## 三、总体方案

### 3.1 核心思路

采用“**员工身份在 AgentNow，智能体实例在 Hermes，知识在共享知识库**”的三层模型：

```text
员工账号（AgentNow User）
    ↓ 1:1 映射
员工 Hermes Profile
    ↓ 绑定
岗位智能体模板（PPT 助手/销售助理/HR 助手）
    ↓ 共享访问
企业公共知识库
```

### 3.2 一句话设计

**每个员工固定绑定一个 Hermes Profile，用 Hermes 原生记忆实现隔离；所有 Profile 通过同一套共享知识库 MCP/RAG 能力获取企业公共知识；AgentNow 负责模板、入口、权限、审计和业务编排。**

### 3.3 推荐架构

```text
前端（员工工作台）
  - 我的智能体
  - 智能体对话页
  - 对话历史
  - 智能体配置查看

后端（AgentNow）
  - Agent 模板管理
  - 用户 -> Hermes Profile 映射服务
  - 会话代理服务
  - 对话历史缓存
  - 审计与权限控制

Hermes（原生）
  - Profile 隔离
  - MEMORY.md / USER.md 记忆
  - API Server / Responses API
  - MCP 工具调用

共享知识库
  - AgentNow 管理文档
  - Hermes 通过 MCP 统一访问
```

### 3.4 从 Hermes WebUI 借鉴的产品思路（不借鉴技术栈）

Hermes WebUI 的最大价值不是“原生 JS / 无构建”，而是它把智能体当成一个工作台，而不是一个聊天框。AgentNow 第一版建议显式吸收以下产品思路：

1. 运行态可见性优先
   - 会话处于哪个阶段、做到了哪一步、哪些确认已经完成，需要在 UI 上可见
   - 生成的文件、版本号、生成状态、失败原因必须可见

2. 消息与动作分离
   - “确认/生成/重试/下载”等属于动作，不应伪装成用户聊天内容
   - 动作要有明确入口，且能回溯到会话状态变更

3. 工作产物是一等对象
   - 文件不是附属信息，而是业务闭环的一部分
   - 历史版本、下载、重试、失败记录都要可追踪

4. 渐进式信息披露
   - 默认界面突出对话与产物
   - 调试信息（检索来源、工具调用摘要、错误堆栈等）可以折叠，但必须可查看

5. “补问/澄清/确认”产品化
   - 在关键节点必须用结构化规则推进，而不是靠长问卷或随机发挥
   - 关键确认点要落库，成为平台可治理状态

---

### 3.5 保持 Hermes 的“越用越聪明”（必须遵守）

Hermes 的核心卖点是“越用越适合一个人”。AgentNow 在做流程产品化时，必须把它做成轨道，而不是笼子。约束如下：

1. 流程控，不语义控
   - AgentNow 只控制“下一步做什么”（需求确认/模板确认/草稿确认/生成文件）
   - 自然语言理解、补问策略、内容生成与修改理解，仍由 Hermes 的模型能力完成

2. Profile 必须稳定
   - 同一员工必须稳定映射到同一个 Hermes Profile，避免每次随机切换导致“越用越聪明”失效

3. Session 必须连续
   - 同一条业务流程（例如一次 PPT 制作）应尽量复用同一 Hermes session 继续推进
   - 避免每轮都新开会话，导致模型反复问同样问题、也无法形成短期偏好

4. 结构化结果是交付载体，不是智能替代
   - 结构化 JSON（例如 `ppt_content`）用于可控渲染、可下载交付、可版本化
   - 但不能用大量硬规则取代模型对用户意图的理解与归纳

5. “压缩成一条最终提示词”要谨慎
   - 可以把关键确认点总结成指令交给 Hermes，但不能丢失用户的修改轨迹与偏好线索
   - 尽量保留对话过程在同一 session 中演进，让 Hermes 自然学习

## 四、如何利用 Hermes 满足需求

### 4.1 每个员工一个 Hermes 智能体

这里不建议“每次发消息临时选一个 Hermes 实例”，而是建议采用 **员工级固定 Profile**。

#### 方案

1. AgentNow 中每个员工对应一个 Hermes Profile
2. Profile 命名规则建议统一：

```text
corp_{tenant_id}_user_{user_id}
```

如果当前系统还是单租户，也建议预留 `tenant_id` 维度，避免未来企业隔离时重构。

#### 好处

1. 直接复用 Hermes 原生记忆隔离能力
2. 每个员工长期拥有自己的上下文和工作风格
3. 便于追踪、审计、运维和恢复
4. 后续可以给同一个员工挂多个业务 Agent，但底层仍可基于同一个 Profile 延续个人记忆

### 4.2 员工与员工之间的记忆隔离

**核心依赖 Hermes Profile 隔离，不建议在 AgentNow 自己维护员工长期记忆。**

#### 隔离机制

1. 员工 A 的所有对话请求只允许路由到自己的 Profile
2. 员工 B 的所有对话请求只允许路由到自己的 Profile
3. Hermes 将各自的 `MEMORY.md`、`USER.md`、对话上下文、技能使用历史保存在各自 Profile 下
4. AgentNow 后端永远不暴露“按用户自由指定 Profile”的员工级接口

#### 后端必须做的控制

1. 员工接口不能接收任意 `profile_name`
2. 所有员工对话接口统一从当前登录用户反查 `user.hermes_profile`
3. 只有超级管理员页面才允许跨 Profile 查看
4. 管理员查看能力与员工使用能力必须走不同接口前缀与权限码

#### 结论

记忆隔离不是“让 AgentNow 不存记忆”这么简单，而是要在**路由层、权限层、服务层**同时保证：员工只能命中自己的 Hermes Profile。

### 4.3 知识库共享

共享知识不要放进员工 Profile 的个人记忆，而要放在**企业公共知识库**。

#### 推荐实现

1. AgentNow 继续负责知识文档上传、分类、元数据、权限和审计
2. Hermes 统一通过已接入的知识库 MCP 服务访问同一个目录
3. 所有员工 Profile 使用同一套知识库连接配置
4. 智能体在回答时自主检索共享知识库，而不是把文档复制进每个 Profile

#### 为什么不能放到 MEMORY.md

1. 容量不够
2. 更新成本高
3. 会在员工之间产生知识版本不一致
4. 公共知识和个人偏好会混在一起，后续难维护

### 4.4 岗位智能体如何落地

Hermes 原生支持不同 Profile 的角色配置，但企业场景更适合由 AgentNow 再增加一层“**Agent 模板**”。

这里要明确一个边界：

1. 第一版的 Agent 模板，**不是单纯一个 Prompt**
2. 第一版的 Agent 模板，**也不是完整的低代码工作流引擎**
3. 第一版更适合定位成“**有一定配置能力的智能体模板平台**”

也就是说，模板本质上是一份“可运营、可配置、可演进”的配置包，先把 80% 的通用能力做成后台可配，先跑通业务闭环，再为后续演进到更强的流程编排能力留接口。

#### 第一版模板的推荐定义

第一版建议把模板定义为以下几类配置的组合：

1. 模板基础信息
2. 角色说明
3. 系统指令
4. 开场白
5. 知识范围
6. 工具权限
7. 输出格式约束
8. 关键交互规则
9. 模板版本
10. 配置快照

#### 模板负责什么

1. 定义智能体“是谁”
2. 定义智能体“主要解决什么问题”
3. 定义回答风格、语气和输出形式
4. 定义是否使用知识库、允许访问哪些知识分类
5. 定义是否允许使用网页搜索、文件生成、PPT 生成等工具
6. 定义遇到关键信息不足时要先补问，而不是直接给结果
7. 定义在生成正式产物前，是否要求用户确认

#### 模板不负责什么

1. 不直接替代底层工具
2. 不直接替代 Hermes 的推理能力
3. 不在第一版中承载复杂状态机和通用流程编排引擎
4. 不追求让管理员在后台拼出任意复杂工作流

#### 模板、Skill、Tool 的关系

为了避免后续理解混乱，建议统一按下面方式定义：

1. **模板**：定义一个岗位智能体怎么工作，是总控配置层
2. **Skill**：可选的专业方法模块，用来提升某类任务的生成质量
3. **Tool**：真正执行动作的能力，比如搜索资料、读取文档、生成文件、导出 PPT

第一版项目里，模板是主角；Skill 可以先少量使用甚至先不做；Tool 必须优先接好。

#### 模板平台的后台化目标

超级管理员应当能在后台完成下面这些动作，而不依赖开发改代码：

1. 新增一个模板
2. 修改模板名称、描述和适用岗位
3. 修改系统提示词和欢迎语
4. 配置知识范围
5. 配置可用工具白名单
6. 配置输出格式要求
7. 启用或停用模板
8. 选择模板更新是只影响新用户，还是同步给已开通员工

这已经可以支撑一个“有一定配置能力的智能体模板平台”。

#### 为后续工作流演进预留空间

虽然第一版不做完整的低代码工作流编排，但建议在模型设计上提前预留两个扩展位：

1. `interaction_rules`
   - 用来描述关键交互规则
   - 例如“资料不足先补问”“大纲先确认再继续”“生成正式 PPT 前再次确认”

2. `workflow_hints`
   - 用来描述轻量流程提示
   - 例如“先需求澄清，再搜资料，再生成大纲，再输出正式稿”

这样做的好处是：

1. 第一版仍然是配置平台，复杂度可控
2. 第二版如果要升级成状态机或流程引擎，不需要推翻表结构
3. PPT 助手、周报助手、售前方案助手等模板可以共用同一套演进路径

#### 推荐落地方式

1. 模板定义岗位身份、职责边界、输出风格、工具使用策略和关键交互规则
2. 员工开通某个智能体时，将模板配置快照绑定到员工智能体记录
3. 对话时由 AgentNow 组装模板配置，再传给 Hermes 执行
4. 模板更新后，可选择：
   - 只影响新开通员工
   - 或由管理员触发同步到已有员工

#### 模板中建议包含

1. 智能体名称
2. 场景说明
3. 系统指令/基础 Prompt
4. 是否启用知识库
5. 是否启用 PPT 相关工具
6. 推荐开场白
7. 输出格式约束
8. 安全边界
9. 关键交互规则
10. 轻量流程提示

### 4.5 关于 PPT 最终文件的实现边界

需要特别说明：如果用户说“帮我制作一个公司介绍的 10 页 PPT”，并且目标是得到一个真正可下载的 `.pptx` 文件，那么仅靠模板和提示词是不够的。

第一版必须遵守下面这个分工：

1. 模板负责定义工作方式
2. Hermes 负责澄清需求、查资料、组织内容、生成结构化结果
3. Tool 负责真正生成最终文件

也就是说，`PPT 助手` 第一版应该采用下面的模式：

```text
模板配置
-> Hermes 生成 PPT 结构化内容
-> AgentNow 调用 PPT 生成服务
-> 输出最终 .pptx 文件
```

因此，PPT 助手不是“纯聊天模板”，而是“模板 + 内容生成 + 文件生成工具”的组合能力。

### 4.6 当前版本的产品定位

结合本次讨论，第一版产品定位建议正式定为：

**AgentNow 智能体配置模板平台（可配置、可运营、可演进）**

不把第一版目标定成“弱代码工作流引擎”，原因有三点：

1. 目标过大会显著提高首版复杂度和失败率
2. 当前最重要的是先把单模板、单案例、单闭环做稳定
3. 先把模板平台跑通，后面再升级成更强的流程平台成本更低

第一版能力边界建议如下：

1. 支持后台新增与编辑智能体模板
2. 支持配置知识范围、工具权限、提示词和交互规则
3. 支持模板版本和快照
4. 支持按模板为员工开通智能体
5. 支持围绕模板跑通业务闭环

暂不纳入第一版范围的内容：

1. 通用流程节点编排器
2. 图形化流程设计器
3. 复杂状态机驱动
4. 通用分支引擎
5. 任意业务流程的零代码拼装

但在实现时，要保留向这些能力演进的空间。

## 五、数据模型设计

在不破坏现有用户表的前提下，第一版建议新增 6 张智能体核心表。

这里要特别说明：**数据库最终落地以 `backend/data/database.sql` 为准，本章节主要用于理解业务关系和设计意图。**

### 5.1 `agent_templates`

用于管理企业可用的智能体模板。

建议字段：

| 字段 | 说明 |
|------|------|
| `id` | 主键 |
| `code` | 模板编码，如 `ppt_assistant` |
| `name` | 模板名称 |

## 六、核心流程设计

### 6.1 员工首次开通智能体

```text
员工进入“我的智能体”
-> AgentNow 检查 user_agents 是否存在
-> 若不存在，选择默认模板（如 PPT 助手）
-> 检查 users.hermes_profile 是否存在
-> 若不存在，自动创建 Hermes Profile
-> 为该 Profile 写入基础配置
-> 建立 user_agents 记录
-> 返回智能体工作台
```

### 6.2 员工发起对话

```text
前端提交消息
-> 后端根据当前登录用户获取 user_agent
-> 读取绑定的 hermes_profile
-> 组装系统指令、模板配置、知识库策略
-> 调用 Hermes Responses API
-> Hermes 按 Profile 使用个人记忆，并在需要时调用共享知识库
-> 后端把会话摘要缓存到本地表
-> 前端展示响应
```

### 6.3 管理员更新模板

```text
管理员修改模板
-> 保存新版本
-> 选择“仅新开通生效”或“同步到现有员工”
-> 若同步
-> 后端遍历 user_agents
-> 将新的模板快照写入记录
-> 视需要同步 Hermes Profile 配置
```

### 6.4 知识库更新生效

```text
管理员上传/更新知识文档
-> AgentNow 写入共享知识目录
-> MCP 索引更新
-> 所有员工智能体后续查询自动使用最新知识
```

## 九、记忆隔离设计细则

### 9.1 哪些信息进入个人记忆

1. 员工表达偏好

## 十一、首个案例：PPT 助手

### 11.1 业务目标

PPT 助手的目标不是直接替代员工做完整设计，而是帮助员工高效完成以下工作：

1. 明确汇报目标
2. 生成大纲
3. 生成每页标题与要点
4. 从企业知识库抽取公司材料、产品卖点、案例数据
5. 输出可直接粘贴到 PPT 的页面文案

### 11.2 目标用户

1. 销售
2. 方案顾问
3. 产品经理
4. 运营
5. 管理层助理

### 11.3 模板定义

模板编码建议：`ppt_assistant`

模板名称建议：`企业 PPT 助手`

第一版不把 `ppt_assistant` 做成通用流程引擎，而是做成一个“可配置模板 + 明确交互规则 + 最终文件生成能力”的业务模板。

建议这个模板至少包含下面这些后台配置项：

1. 模板名称与说明
2. 系统提示词
3. 欢迎语
4. 可用知识分类
5. 是否允许外部网页搜索
6. 是否启用 PPT 文件生成
7. 输出格式要求
8. 用户确认规则
9. 模板版本号

#### 系统指令建议

```text
你是企业内部的 PPT 助手，负责帮助员工快速完成汇报类、方案类、介绍类 PPT 的策划与文案输出。

你的工作原则：
1. 先判断本次 PPT 的目标、对象、时长、场景，再输出内容。
2. 涉及公司信息、产品能力、案例数据、品牌表述时，优先查询企业知识库。
3. 如果知识库没有明确依据，要主动说明该部分为通用建议，不得编造企业事实。
4. 默认输出结构化结果，包括：封面标题、目录、逐页标题、每页要点、演讲提示。
5. 输出风格简洁、商务、适合直接粘贴到 PPT 页面。
6. 当用户信息不足时，先补问关键问题，不直接盲写整套 PPT。
```

### 11.4 首期支持的对话能力

1. “帮我写一个客户拜访汇报 PPT 大纲”
2. “把这个主题扩成 10 页 PPT”

## 十三、小步快跑的落地路径

这一章专门说明：第一版不要试图一次把“模板平台、PPT 工作流、知识检索、最终文件生成、后台运营”全部同时做完，而要按小步快跑的方式推进。每一步都追求“范围小、可验证、可回退、可复盘”。

### 第一步：先打通一个员工、一个模板、一次对话

#### 这一阶段要实现什么

1. 后台先只有一个模板：`ppt_assistant`
2. 一个员工首次进入时，可以自动绑定 Hermes Profile
3. 员工可以进入“我的智能体”并发起第一轮对话
4. 先不生成 `.pptx` 文件，只要求能稳定输出文字版大纲和逐页建议

#### 这一阶段为什么先做这个

因为这一阶段的目标只有一个：先验证“员工账号 -> Hermes Profile -> 模板配置 -> 对话输出”这条链路是通的。

#### 这一阶段模板提示词建议

```text
你是企业内部的 PPT 助手。

你的第一任务不是直接写完整 PPT，而是先弄清楚用户要做什么。

如果用户的信息不完整，请主动补问这些内容：
1. 这份 PPT 是给谁看的
2. 用在什么场景
3. 预计做多少页
4. 希望偏正式、偏销售，还是偏介绍型

在信息不完整时，不要直接生成整套 PPT，只先给一个初步建议或先提问。
```

#### 这一阶段完成标准

1. 员工能进入自己的智能体
2. 系统能稳定返回回答
3. 不同员工不会串到同一个 Profile
4. 管理员能看到员工已开通的模板和 Profile

### 第二步：让智能体先学会“先问清楚，再出大纲”

#### 这一阶段要实现什么

1. 增加补问逻辑
2. 支持输出更稳定的 PPT 大纲结构
3. 增加“大纲确认”这个关键交互
4. 用户确认前，不进入正式内容生成

#### 这一阶段的核心思路

这一阶段先不追求最终文件，而是把交互过程做对。因为真正影响使用体验的，往往不是 PPT 文件导出，而是智能体有没有把事情问清楚、有没有先给出合理大纲。

#### 这一阶段模板提示词建议

```text
当用户提出做 PPT 的需求时，你要按下面顺序工作：

第一，确认主题、对象、场景、页数和风格。
第二，根据用户已提供的信息，先生成一个清晰的大纲。
第三，把大纲分成逐页标题和每页 2 到 4 条核心要点。
第四，在用户明确确认之前，不要进入正式成稿阶段。

如果用户说“先给我看大纲”，你就只输出大纲，不要把每页写得过长。
```

#### 这一阶段完成标准

1. 系统能稳定输出 8 到 12 页大纲
2. 大纲结构清晰，适合直接确认
3. 用户确认后，系统才继续扩写
4. 用户不确认时，系统支持回改大纲

### 第三步：接入企业知识库，让内容更像“公司自己的 PPT”

#### 这一阶段要实现什么

1. 模板支持配置知识范围
2. PPT 助手能够优先检索公司介绍、产品资料、案例材料、品牌规范
3. 输出内容中尽量引用企业已有资料，而不是纯通用表达
4. 没有依据的内容，要明确说明是通用建议

#### 这一阶段的核心思路

先把“能说”升级成“说得像公司自己的内容”。如果没有这一步，PPT 助手的价值会比较像通用大模型，而不是企业智能体。

#### 这一阶段模板提示词建议

```text
当任务涉及公司介绍、产品能力、案例数据、品牌表达时，请优先查询企业知识库。

如果知识库中已经有明确资料，请优先采用资料中的表述方式，不要随意改写核心事实。

如果知识库没有足够依据，请直接告诉用户这部分先提供通用建议，不要编造企业信息。

输出时尽量区分：
1. 可以直接用于正式材料的企业信息
2. 需要用户确认的通用补充建议
```

#### 这一阶段完成标准

1. 输出中能体现企业资料
2. 用户能明显感受到内容更贴近公司
3. 资料不足时不会乱编
4. 管理员可以通过后台调整知识范围

### 第四步：增加模板配置能力，让后台可运营

#### 这一阶段要实现什么

1. 增加模板管理页面
2. 超级管理员可修改模板名称、说明、提示词、欢迎语
3. 可配置知识范围、工具权限、输出格式要求
4. 模板支持版本和快照
5. 模板修改后可选择只影响新员工，或同步到已有员工

#### 这一阶段的核心思路

从这一阶段开始，系统就不只是“写死的 PPT 助手”，而是具备了“模板平台”的雏形。虽然还不是低代码工作流平台，但已经能让业务人员通过后台持续运营模板。

#### 这一阶段模板提示词建议

这一阶段不是新增某一个固定 Prompt，而是要允许管理员在后台修改下面这些自然语言配置：

```text
角色说明：
你是企业内部的 PPT 助手，擅长公司介绍、售前宣讲、项目汇报和管理层汇报材料。

工作原则：
先问清需求，再组织资料，再给出大纲，再进入正式内容。

知识使用要求：
涉及公司事实时优先使用知识库资料。

输出要求：
先短后长，先结构后细节，先确认后定稿。
```

#### 这一阶段完成标准

1. 管理员可在后台修改模板
2. 员工侧能感知模板变化
3. 旧会话不受新模板直接污染
4. 同一个平台可开始准备第二个模板

### 第五步：让 PPT 助手真正生成最终文件

#### 这一阶段要实现什么

1. 新增 PPT 生成服务
2. 支持把结构化内容渲染成 `.pptx`
3. 支持选择企业标准 PPT 母版
4. 用户确认大纲和风格后，生成正式文件
5. 返回下载地址或预览入口

#### 这一阶段的核心思路

这一阶段是从“内容助手”升级为“交付助手”的关键一步。只有把 `.pptx` 文件真正生成出来，PPT 助手才算完成闭环。

#### 这一阶段模板提示词建议

```text
当用户已经确认大纲和输出风格后，请把内容整理成适合生成 PPT 的结构化结果。

每一页都需要给出：
1. 页面标题
2. 页面要点
3. 可选副标题
4. 演讲备注

在生成正式文件前，请再次确认：
1. 页数是否确认
2. 风格是否确认
3. 是否使用公司标准模板

确认后，再进入最终 PPT 生成。
```

#### 这一阶段完成标准

1. 用户可以拿到 `.pptx` 文件
2. 文件结构正确，至少能正常打开和编辑
3. 页面内容与确认的大纲一致
4. 生成失败时可以重试并看到错误信息

### 第六步：补齐预览、修改、重试和运营能力

#### 这一阶段要实现什么

1. 支持查看已生成 PPT 的记录
2. 支持对已生成结果进行二次修改
3. 支持失败重试
4. 增加模板使用统计和生成成功率统计
5. 增加管理员问题排查入口

#### 这一阶段的核心思路

这一阶段不追求新能力，而是让系统真正稳定、可运营。很多第一版系统不是死在“做不出来”，而是死在“出了问题没人知道怎么修”。

#### 这一阶段模板提示词建议

```text
当用户要求修改已生成的 PPT 时，请先判断修改范围。

如果只是调整个别页面，请优先做局部修改，不要整份重写。

如果用户要求“整体更正式”“更偏销售”“更偏管理层”，请保留原有结构，重点调整表达方式和页面重心。

如果关键信息变化过大，请建议重新生成大纲后再制作正式版。
```

#### 这一阶段完成标准

1. 用户可以基于已有结果继续迭代
2. 管理员能看到主要问题和失败原因
3. 模板的使用效果可被量化
4. 系统具备初步稳定运营能力

### 第七步：为后续演进到工作流平台预留接口

#### 这一阶段要实现什么

1. 在模板表中预留交互规则字段
2. 记录关键确认节点
3. 为后续步骤状态流转保留结构
4. 让 PPT 助手的关键步骤可被显式记录

#### 这一阶段的核心思路

这一阶段不要求真正做成低代码流程编排，但要在设计上留下演进空间，避免后面全部重做。

#### 这一阶段建议保留的信息

1. 当前智能体在哪个关键阶段
2. 用户是否已经确认大纲
3. 用户是否已经确认模板
4. 当前生成的是草稿还是正式版
5. 本次会话用了哪些知识来源和工具

#### 这一阶段完成标准

1. 当前版本继续保持简单可用
2. 后续升级为更强流程平台时不需要推翻现有模板模型
3. PPT 助手成为后续多模板扩展的参考样板

   - 发布模板

   - 查看 Profile 映射异常

## 十五、风险与应对

| 风险 | 说明 | 应对 |
|------|------|------|
| Hermes API 能力与预期不完全一致 | 实际可用接口可能与文档存在差异 | 第一阶段先做 POC，优先打通单 Profile 对话 |
| 每员工独立 gateway 成本高 | 员工数上来后端口与进程管理复杂 | MVP 先支持小规模，后续再评估集中式接入 |
| 知识库检索效果不稳定 | 命中文档不准会影响业务体验 | 为 PPT 助手先做高质量资料整理与分类 |
| 模板更新影响已有用户体验 | 同步策略不清晰会导致结果波动 | 使用模板快照机制 |
| 员工越权查看他人数据 | 若接口设计不严会泄漏记忆和对话 | 员工接口禁止传入任意 Profile 名称 |

## 十六、验收标准

### 16.1 平台能力验收

1. 员工首次进入可以自动创建并绑定 Hermes Profile
2. 员工只能访问自己的对话与记忆上下文
3. 管理员可以查看模板、开通情况与异常状态
4. 知识库更新后，多个员工智能体都能检索到最新内容

### 16.2 PPT 助手验收

1. 输入一个主题后，能先补问必要信息或直接生成 PPT 大纲
2. 能生成逐页标题与要点
3. 涉及企业信息时，优先引用知识库内容
4. 对无依据内容能明确说明是通用建议
5. 员工 A 与员工 B 连续对话后，彼此风格偏好不会串线

---

## 十七、建议的开发顺序

按当前项目状态，建议不要一上来就做“多模板、多岗位、多管理功能”，而是按下面顺序推进：

1. **先打通单模板**
   - 默认就一个 `ppt_assistant`

2. **先打通单用户闭环**
   - 创建 Profile
   - 发送消息
   - 保存历史

3. **再打通多用户隔离**
   - 做权限和路由收口
   - 验证不同员工记忆不会串

4. **再补模板管理**
   - 支持更多岗位模板

5. **最后补运营与审计**
   - 统计、日志、修复工具

---

## 十八、最终建议

### 18.1 推荐决策

1. **底层实例模型**：每员工一个 Hermes Profile
2. **个人长期信息**：放 Hermes 记忆
3. **企业公共资料**：放共享知识库
4. **业务角色差异**：放 AgentNow 模板层
5. **首个案例**：优先做 PPT 助手

### 18.2 为什么这样最合适

因为这个方案同时满足了四件事：

1. 最大化复用 Hermes 原生能力
2. 最小化对现有 AgentNow 架构的冲击
3. 最容易做出首个可演示闭环
4. 后续从一个案例扩到多个岗位也最顺滑

## 十九、下一步落地清单

建议你们下一步直接进入开发的事项如下：

1. 以 `backend/data/database.sql` 为准，先落地 6 张智能体核心表
2. 先完成后端 `agent`、`agent_template`、`hermes_profile` 基础服务骨架
3. 先实现“员工首次开通默认 PPT 助手”
4. 接 Hermes Responses API 打通单轮/多轮对话
5. 做员工侧“我的智能体 + 对话页”
6. 准备 PPT 助手首批知识资料并接入知识范围控制
7. 再补模板管理页和模板版本能力
8. 最后接入 PPT 文件生成服务
