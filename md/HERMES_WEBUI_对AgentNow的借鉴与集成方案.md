# Hermes WebUI 对 AgentNow 的借鉴与集成方案

## 一、本文目的

这份文档专门回答下面几个问题：

1. `hermes-webui` 对 `AgentNow` 到底有没有借鉴价值
2. 在 `AgentNow` 的 `Hermes管理` 下增加一个 iframe 入口，把 `hermes-webui` 嵌进去，这件事是否可行
3. 如果可行，应该按什么原则做，才能不破坏 `AgentNow` 现有整体架构和思路
4. 除了 iframe 嵌入之外，`AgentNow` 还可以从 `hermes-webui` 借鉴哪些实现模式
5. 后续如果对 `AgentNow` 做一轮“借鉴式重构”，边界应该怎么控制

这份文档的基本前提是：

- `AgentNow` 当前整体架构和产品思路不改
- 不推翻现有前后端分层
- 不把 `AgentNow` 变成 `hermes-webui` 的翻版
- 只做“吸收优点、补足短板、增强系统”的借鉴式升级

---

## 二、先说结论

我的结论是：

### 2.1 `hermes-webui` 对 AgentNow 很有借鉴价值

它最有价值的不是技术框架本身，而是下面这些东西：

- Hermes 原生能力的组织方式
- Profile、会话、工作区、审批、澄清、记忆、技能、MCP 等能力的产品化方式
- Agent 控制台式交互方式
- 流式事件驱动和多面板联动的思路
- 对“智能体不是纯聊天框，而是工作台”的理解

### 2.2 直接 iframe 嵌入“理论上可做”，但“不是拿来就能嵌”

主要原因是：

- `hermes-webui` 默认返回 `X-Frame-Options: DENY`
- 它带有较严格的安全头
- 它本身有自己的登录、Cookie、会话和导航体系

所以：

- 从浏览器前端角度，不能直接随便 `<iframe src="...">` 就完事
- 必须先解决安全头和认证问题
- 还要明确它在 `AgentNow` 里扮演什么角色

### 2.3 推荐把 iframe 嵌入定位成“补充能力”，不是“主能力”

更合适的角色是：

- 在 `Hermes管理` 下增加一个“WebUI 工作台”或“Hermes 控制台”入口
- 作为高级运维、调试、排障、深度操作入口
- 主要给管理员使用

而不是：

- 让它取代 `AgentNow` 现有管理页面
- 让员工侧直接进入
- 把 `AgentNow` 现有的数据闭环和权限体系全部交给它

### 2.4 对 AgentNow 的重构建议应该是“借鉴模式”，不是“借鉴技术栈”

我的判断很明确：

- 应该借鉴 `hermes-webui` 的产品组织方式、交互方式、状态管理思路
- 不应该把 `AgentNow` 重构成 `http.server + 原生 JS + 本地 JSON`

因为 `AgentNow` 的方向是企业平台，和 `hermes-webui` 的单机自托管控制台定位不同。

## 三、为什么这个项目值得 AgentNow 借鉴

## 3.1 它更贴近 Hermes 的“原生工作方式”

`AgentNow` 当前的 Hermes 管理已经做了不少页面和接口，但本质上还是“平台式管理界面”。

而 `hermes-webui` 做得更像一个真正的 Hermes 控制台，它把这些能力连接得更自然：

- 会话
- Profile
- Workspace
- 文件预览
- 工具调用
- 审批
- 澄清
- 技能
- 记忆
- MCP
- Cron
- 设置

对 `AgentNow` 来说，这意味着一个很重要的启发：

`Hermes管理` 不应该只是“若干独立页面的集合”，还应该逐步演进成一个真正可操作、可联动、可排障的 Hermes 工作台。

## 3.2 它对“智能体工作台”的理解比普通后台更成熟

`hermes-webui` 的一个核心价值是，它不是把智能体看成“回答框”，而是看成“带上下文、工具、文件、工作区和长期记忆的工作系统”。

这对 `AgentNow` 非常有参考意义，尤其是你现在正在做智能体核心功能。

可借鉴的点包括：

- 聊天不是唯一主体，工作区和文件协作也很重要
- 审批和澄清应该是标准能力，不是特殊补丁
- 会话、Profile、记忆、工具调用应该能串起来看
- 管理员需要的不只是“查配置”，而是“看运行现场”

## 3.3 它能帮助 AgentNow 的 Hermes管理更快补齐后半段能力

从当前 `AgentNow` 代码来看，`Hermes管理` 的部分能力已经完成，但也还有一部分仍在开发中，或者是基础版。

例如当前已有：

- 系统概览
- Profiles 管理
- 技能、MCP、记忆、配置、审计等路由和页面骨架

但也能看出：

- 有些页面还比较轻
- 有些只是管理页，不是操作台
- 深度联动和现场感不够强

`hermes-webui` 刚好能提供一个“深度控制台”的补充视角。

## 四、iframe 方案可行性分析

## 4.1 方案定义

你当前的想法是：

- 在 `AgentNow` 的 `Hermes管理` 下面新增一个功能点
- 用 `iframe` 方式把 `hermes-webui` 嵌进去

这个想法本质上是：

在不大改 `AgentNow` 架构的前提下，快速把 Hermes 原生 Web 控制台能力接入进来。

这个方向本身是合理的，而且很务实。

## 4.2 可行性判断

我的判断是：

- 从产品方向上，可行
- 从技术实现上，可行
- 但从安全和集成细节上，不是“直接可用”，需要做一层受控接入

也就是说，这不是一个伪需求，而是一个值得做的集成方向。

## 4.3 当前最大的技术阻碍

`hermes-webui` 默认会给所有响应加安全头：

- `X-Frame-Options: DENY`

其实现位置在：

- [helpers.py](file:///Users/victor/code/meetchances/AgentNow/AgentNow/hermes-webui/api/helpers.py#L38-L55)

这意味着：

- 任何页面默认都不允许被 iframe 嵌套
- 就算是同域，也不允许

所以，如果完全不处理这一层，`iframe` 方案是行不通的。

## 4.4 除了安全头，还有几个实际问题

### 4.4.1 认证问题

`hermes-webui` 有自己的一套登录和 Cookie 机制。

如果直接嵌入，会遇到：

- 用户是否要二次登录
- 登录态是否和 `AgentNow` 打通
- Cookie 域和路径如何处理
- 登录过期后，iframe 里如何回到登录页

### 4.4.2 导航重复问题

`AgentNow` 本身已经有自己的顶部导航和页面结构。

而 `hermes-webui` 也有自己的导航、控制中心和工作台逻辑。

如果直接全量嵌入，会出现：

- 外层一套导航
- iframe 里面又一套导航
- 视觉上容易像“系统里套系统”

### 4.4.3 权限边界问题

`AgentNow` 当前有明确的员工侧和管理端权限体系。

而 `hermes-webui` 更偏“直接操作 Hermes 实例”的工作台。

所以一定要限制：

- 只能在 `Hermes管理` 下使用
- 只对管理员开放
- 不能让员工侧直接借这个入口绕过平台权限

### 4.4.4 状态割裂问题

如果 iframe 只是“嵌一个独立系统”，那用户会觉得：

- `AgentNow` 的 Hermes 页面是一套
- `hermes-webui` 里面又是一套
- 两边数据和操作上下文不统一

这会导致体验上割裂。

所以必须提前定义好：

- 哪些功能以 `AgentNow` 为主
- 哪些功能以 `hermes-webui` 为辅

## 五、我对 iframe 方案的建议结论

## 5.1 不建议直接“全量主页嵌入”

不建议一上来就把整个 `hermes-webui` 首页完整塞进 `AgentNow`。

原因：

- 安全头要处理
- 认证复杂
- 双层导航混乱
- 会削弱 `AgentNow` 自己的产品一致性

## 5.2 建议做成“单独入口页”

更合理的方式是：

在 `Hermes管理` 下新增一个二级菜单，例如：

- `Hermes 控制台`
- `WebUI 工作台`
- `Hermes Web 控制台`

点击后进入一个独立页面，这个页面主要内容就是嵌入的 `iframe`。

这样做的好处是：

- 产品边界清晰
- 用户心理预期清晰
- 不会污染现有 Hermes 概览、Profiles、技能、审计等页面

## 5.3 建议把它定位为“高级入口”

这个入口更适合承担：

- 深度调试
- 原生 Hermes 操作
- 复杂工作区交互
- 深层会话现场查看
- 临时运维与排障

不建议承担：

- 企业级主数据管理
- 员工权限控制
- AgentNow 的业务主流程

也就是说：

`AgentNow` 负责平台治理和企业封装，`hermes-webui` 负责 Hermes 原生操作台补充。

## 5.4 建议只给管理员开放

这件事我认为要写成明确原则：

- 员工侧不开放 iframe 版 Hermes WebUI
- 只在 `Hermes管理` 下开放
- 只给系统管理员 / 超级管理员使用

原因很简单：

- 这个入口偏底层、偏原生、偏运维
- 不适合普通员工直接用
- 也避免破坏员工侧智能体产品体验

## 六、如果要做 iframe 集成，实施原则应该是什么

## 6.1 原则一：只做补充，不做替代

`hermes-webui` 进入 `AgentNow` 后，只能是补充能力，不能替代现有平台能力。

明确边界：

- `AgentNow` 仍然是主平台
- `AgentNow` 的数据库、权限、审计、模板、员工智能体体系仍然是主事实来源
- `hermes-webui` 只是 Hermes 原生控制台能力的增强入口

## 6.2 原则二：不修改 AgentNow 当前核心架构

不能因为这个项目很轻，就把 `AgentNow` 往那个方向整体迁移。

`AgentNow` 应继续保持：

- 现有前端工程结构
- 现有后端服务分层
- 现有数据库与权限体系
- 现有员工侧 / 管理端边界

借鉴的是“模式”，不是“推翻重来”。

## 6.3 原则三：优先同域代理，不建议直接跨域裸嵌

如果后续真的实现 iframe，推荐方式不是直接嵌一个外部地址，而是：

- 通过统一域名或反向代理路径暴露 `hermes-webui`
- 例如挂到 `AgentNow` 的某个受控路径下

这样好处包括：

- 更容易统一认证
- 更容易处理 Cookie 和登录态
- 更容易做访问控制
- 更方便隐藏内部部署细节

当然，这一层通常需要配合：

- Nginx / 网关
- 或后端代理转发

## 6.4 原则四：安全优先

iframe 集成一定要把安全放第一位。

至少要关注：

- `X-Frame-Options` / CSP 的处理
- 登录态隔离
- 管理员身份校验
- 防止未授权访问 Hermes 原生控制台
- 审计 iframe 入口访问

这件事不能只是前端加个菜单。

## 6.5 原则五：尽量减少双重导航干扰

如果嵌入的是完整 `hermes-webui`，最好考虑：

- 是否可以以后增加“嵌入模式”
- 让其在 iframe 模式下弱化自身导航或头部

如果暂时做不到，也应该至少在 `AgentNow` 侧把页面定位讲清楚：

- 这是“原生 Hermes 控制台”
- 不是 `AgentNow` 普通管理页

## 6.6 原则六：入口要清晰标注边界

这个页面建议带清晰说明，例如：

- 当前页面嵌入的是 Hermes 原生 Web 控制台
- 适用于高级运维、调试和深度查看
- 企业级主配置、员工和模板仍以 AgentNow 页面为准

这样可以减少误用和认知混乱。

## 七、我建议的集成方式

## 7.1 推荐方案：分层集成

### 第一层：保留 AgentNow 现有 Hermes管理页面

这些页面继续存在，而且继续作为主入口：

- 系统概览
- Profiles 管理
- 对话日志
- 技能管理
- MCP 服务
- 记忆系统
- 配置管理
- 审计

这些是企业管理和平台治理层页面，不应该被替换。

### 第二层：新增 Hermes WebUI 工作台入口

在 `Hermes管理` 下新增一个菜单：

- `WebUI 工作台`

功能上：

- 单独路由
- 页面主体是 iframe
- 可加一些顶部提示和快捷说明

它承担“深度原生操作”能力。

### 第三层：后续逐步把真正高价值的能力再平台化

这一步不是现在立刻做，而是后续逐步判断：

- 哪些能力放在 iframe 中就够了
- 哪些能力值得沉淀到 `AgentNow` 自己的页面与接口中

例如：

- 日常治理能力，适合平台化
- 深度调试能力，适合保留原生控制台

## 八、hermes-webui 对 AgentNow 的重构借鉴价值

这里说的“重构”，不是推翻架构，而是借鉴其更成熟的组织方式。

## 8.1 可以借鉴的第一类：Hermes 能力组织方式

当前 `AgentNow` 的 Hermes 管理更偏“模块式后台”。

而 `hermes-webui` 给出的启发是：

- 要把 Profile、会话、工作区、文件、记忆、技能、工具调用、审批等能力串成工作链

对 `AgentNow` 来说，可借鉴的方向是：

- 让 Hermes 管理模块之间产生联动
- 从“静态页面集合”升级为“运行态工作台”

## 8.2 可以借鉴的第二类：智能体交互工作台思路

`hermes-webui` 很强调：

- 会话上下文
- 工具执行现场
- 审批 / 澄清
- 文件协作
- 运行态可视化

这些思路对 `AgentNow` 员工侧智能体后续也很重要。

尤其后面如果你继续增强智能体系统，完全可以借鉴：

- 更清晰的阶段感
- 更完整的消息与动作分离
- 工具调用与结果的可视化
- 会话与文件、工作区的联动

## 8.3 可以借鉴的第三类：流式事件与实时状态设计

`hermes-webui` 大量使用 SSE，除了聊天，还用于：

- approval
- clarify
- gateway 状态

这对 `AgentNow` 后续很有启发。

`AgentNow` 当前为了先把前 5 步做稳，采用了非流式方案，这是对的。

但后续可以借鉴：

- 在不破坏现有结构的前提下，逐步补流式能力
- 先从管理员调试页、复杂会话页开始
- 再考虑员工侧智能体增强体验

## 8.4 可以借鉴的第四类：系统级运行态可见性

`hermes-webui` 的强项之一是“现场感”。

用户能比较直观地看到：

- 现在在哪个会话
- 当前用哪个 Profile
- 文件和工作区是什么
- 工具执行了什么
- 系统状态怎么变化

这对 `AgentNow` 的借鉴价值非常大。

因为企业级平台最怕的是：

- 有配置
- 有接口
- 有页面
- 但没有“运行现场可见性”

## 8.5 可以借鉴的第五类：首次引导和环境诊断思路

`hermes-webui` 的 `bootstrap + onboarding` 思路很值得参考。

对 `AgentNow` 可借鉴的不是“照抄启动器”，而是：

- 把系统依赖检查做成产品能力
- 把环境缺失、配置异常、Profile 不一致等问题前置暴露
- 管理页不要只显示结果，也要显示诊断过程

## 九、哪些东西不应该借鉴

为了避免走偏，我认为下面这些内容不应该借鉴为 `AgentNow` 的重构方向。

## 9.1 不要借鉴它的后端技术栈

`hermes-webui` 使用：

- Python 标准库 HTTP Server
- 本地 JSON 文件状态

这很适合单机自托管小系统，但不适合 `AgentNow` 这种企业平台方向。

`AgentNow` 应继续坚持：

- 标准后端框架
- 数据库存储
- 明确权限体系
- 企业级数据与审计链路

## 9.2 不要借鉴它的“单机默认假设”

`hermes-webui` 的很多实现默认：

- 单用户
- 轻并发
- 本地文件即数据库

而 `AgentNow` 不是这个定位。

## 9.3 不要借鉴“一个大路由文件做所有事”

`hermes-webui` 的 `api/routes.py` 很强，但不适合在 `AgentNow` 继续复制这种模式。

`AgentNow` 当前更适合继续走：

- router
- service
- model
- schema

的清晰分层。

## 9.4 不要把 iframe 方案变成偷懒替代

iframe 应该是增强，不应该成为：

- 不做管理页的借口
- 不做平台化能力的借口
- 不做权限治理和审计的借口

否则 `AgentNow` 会慢慢变成一个外壳，而不是平台。

## 十、对 AgentNow 的推荐实施原则

这里我把建议收束成一套明确原则，后续无论你自己开发，还是让大模型继续做，都可以按这套原则约束。

### 原则 1：平台主导原则

`AgentNow` 始终是主平台。

- 平台规则
- 权限体系
- 数据主表
- 审计
- 员工侧体验

都必须由 `AgentNow` 主导。

### 原则 2：原生能力补充原则

`hermes-webui` 只补 Hermes 原生深度操作能力。

它更适合作为：

- 运维工作台
- 原生控制台
- 调试入口

### 原则 3：管理侧专用原则

iframe 集成仅限管理侧，不能进入员工侧主流程。

### 原则 4：模式借鉴优先于代码照搬

重点借鉴：

- 交互模式
- 状态组织
- 工作台结构
- 流式事件
- 运行态可见性

不重点借鉴：

- 技术栈本身
- 本地 JSON 存储
- 单机路由风格

### 原则 5：渐进式重构原则

不要一次重构太多。

更合理的顺序是：

1. 先补一篇方案文档
2. 再新增一个 iframe 入口页
3. 再解决安全头与认证接入
4. 再决定哪些能力继续保留在原生页面
5. 最后再逐步借鉴它的交互和工作台组织方式

### 原则 6：避免双重事实来源原则

凡是企业级核心数据，仍以 `AgentNow` 为准：

- 员工
- 角色
- 权限
- 智能体模板
- 员工智能体实例
- 会话业务数据
- 审计记录

不要把这些核心平台数据变成两边各自维护。

## 十一、建议的落地顺序

如果后续真的要推进，我建议按下面顺序做。

### 第一步：只做方案确认，不急着编码

先确认三件事：

1. iframe 入口只给管理员用
2. `hermes-webui` 只是 Hermes 控制台补充入口
3. `AgentNow` 现有 Hermes 管理页继续保留为平台管理主入口

### 第二步：先做一个最小入口页

在 `Hermes管理` 下新增：

- `WebUI 工作台`

先不做复杂打通，只把页面结构和入口预留出来。

### 第三步：解决嵌入技术问题

重点处理：

1. `X-Frame-Options: DENY`
2. 认证打通或受控登录
3. 同域代理或受控反向代理
4. 页面尺寸、自适应和失败提示

### 第四步：明确能力边界

文档写清楚：

- 哪些功能用 `AgentNow`
- 哪些功能用 `hermes-webui`
- 哪些功能未来要从 `hermes-webui` 经验中平台化沉淀

### 第五步：再做借鉴式重构

重构优先级建议：

1. 先增强 Hermes 管理的运行态可见性
2. 再增强智能体交互工作台感
3. 再考虑流式事件和复杂操作台

而不是先去大改技术栈。

---

## 十二、最终建议

如果一句话概括我的建议：

`hermes-webui` 非常值得 AgentNow 借鉴，但应该借鉴它的产品组织方式和运行态工作台思路，而不是照搬它的技术实现；iframe 集成可以做，但必须把它定位为 Hermes 管理下的高级控制台入口，并严格处理安全、认证、权限和边界问题。`

再说得更直接一点：

- 这个方案值得做
- 但不能粗暴做
- 不能让它取代 AgentNow
- 不能因为它很强，就把 AgentNow 的平台化路线打乱

最稳妥的方向是：

- `AgentNow` 继续做企业智能体平台
- `hermes-webui` 作为 Hermes 原生深度操作台的补充入口
- 逐步吸收它在工作台、流式交互、运行态可见性方面的优点

这才是既稳健、又真正有价值的做法。
